# riscv-mini 计算机系统综合实验 — 答辩陈述

---

## 一、项目概览（30 秒电梯演讲）

> 基于 UC Berkeley 开源的 **riscv-mini**（RV32I 三级流水线处理器，Chisel 实现），在 5 组 Cache 微架构参数下完成了 **参数调优实验**，同时自主编写了 **175 行 16×16 矩阵乘法裸机程序**，打通 C/汇编→ELF→hex→Verilator 仿真→性能分析全流程。

---

## 二、做了什么（核心贡献）

### 2.1 Cache 参数调优实验

| 项目 | 内容 |
|------|------|
| 测试配置 | 5 组 (nWays, nSets, blockBytes)：4KB / 8KB / 16KB |
| 测试负载 | 3 个官方 benchmark + 3 个规模的自定义 matrix_mul (N=4/8/16) |
| 采集指标 | CPI、I$/D$ Miss Rate、I$/D$ Stall、Total Cycles |
| 自动化 | Python 脚本自动修改 Config → 重编译 → 仿真 → 汇总 CSV |

**关键发现**：

```
┌─────────────────────────────────────────────────────────┐
│  "性能悬崖"：16KB → 14万cycle 正常完成                    │
│              8KB  → 5000万cycle 超时（354×）仍未退出       │
│              4KB  → 同上                                  │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  块大小翻倍（16B→32B）：Miss 率 ↓50%，但 CPI 不变（1.21）   │
│  → 瓶颈已从 Cache miss 转移至流水线数据/控制冲突            │
└─────────────────────────────────────────────────────────┘
```

### 2.2 自定义矩阵乘法裸机程序

- **175 行** C + RV32I 纯汇编（远超 50 行最低要求）
- 覆盖 RV32I **全部 6 类指令格式**（R/I/S/B/U/J + SYSTEM）
- 软件乘法：移位加法实现 32-bit 乘法（无 M 扩展）
- 三重循环结构：外层 N=16 → 4096 次乘法累加

## 三、怎么做的（技术路线）

### 3.1 工具链全流程

```
                 ┌─ Chisel 源码 ─┐
                 │ Cache.scala   │
                 │ Config.scala  │──→ sbt run ──→ Tile.sv ──→ make verilator ──→ VTile
                 │ Datapath.scala│
                 └───────────────┘
                                           │
main.c + matrix_mul.S ──→ gcc/elf2hex ──→ main.hex ──→ ./VTile main.hex
                                                          │
                                                    stderr: CPI, Miss%, Stall
```

### 3.2 Cache 微架构理解

```
CPU 地址 (32-bit)
├── Tag (tlen) ──── 匹配判断
├── Index (slen) ── 选择 Cache Set（nSets=512 → 9-bit）
└── Offset (blen) ─ 块内字节偏移（16B → 4-bit）

Cache 状态机（7 状态）：
sIdle → sReadCache/sWriteCache → (hit→sIdle) / (miss→sWriteBack→sRefill→sIdle)
```

### 3.3 流水线停顿机制

```scala
// Datapath.scala:78
val stall = !io.icache.resp.valid || !io.dcache.resp.valid
```

- I$ 或 D$ 任一 `resp.valid=0`，整个三级流水线暂停
- **重要**：Cache 模块的 `perf.stall` 计数器统计的是 Cache 状态机忙碌周期（`state ≠ sIdle`），**不是流水线停顿周期**。I$ 几乎每个周期都在取指（sReadCache），因此 I$Stall≈98.7% 总 cycle，但实际流水线停顿仅 17.4%

## 五、遇到的坑 & 解决方案

| 坑 | 现象 | 根因 | 解决 |
|----|------|------|------|
| **数据不变** | 5 组配置 CPI 完全相同 | `make verilator` 未重新链接 VTile | `rm -f VTile` 强制重建 |
| **入口错误** | 程序 8 cycle 退出 | _start 被链接到 0x400 而非 0x200 | test.ld 中 `*crt.o(.text)` 优先 |
| **SSH 超时** | 仿真 2.5min 时 SSH 断开 | TCP keepalive 不足 | nohup + 后台 + 轮询 |
| **计数器为空** | 超时配置性能数据缺失 | top.cc 超时分支未打印计数器 | 分析代码路径，"5000万cycle"本身即为证据 |
| **正则转义** | SSH heredoc 中 `I\$` 退化为 `I$` | bash 双引号消耗反斜杠 | 手动 grep 采集，准确性优先 |

## 六、学到了什么（总结）

1. **"性能悬崖"不是教科书上的抽象概念** — 亲手把 Cache 从 16KB 降到 8KB，眼睁睁看着仿真从 14 万 cycle 飙升到超时未完成，354× 的差距比任何理论分析都震撼

2. **硬件计数器的语义必须从源码确认** — "stall" 在不同模块含义不同：Cache.stall = 忙碌周期，Datapath.stall = 流水线停顿。轻信命名会导致分析完全错误

3. **工程可复现性是实验的生命线** — Makefile 隐式规则、Shell 转义、SSH 超时，每个细节单独看都很小，但合在一起可能让你怀疑人生

4. **软硬件协同视角** — 写 matrix_mul 汇编时同时思考 Cache 行为、流水线停顿、ABI 约定，这种穿透抽象层次的思考是计算机系统课程的核心价值

## 七、Q&A 预备

| 可能的问题 | 准备好的回答 |
|-----------|-------------|
| **为什么 CPI 只有 1.21？三级流水线理想 CPI 不是 1.0 吗？** | 主要来自：① Cache miss 停顿（~2K-3K cycles）② 数据冒险（load-use hazard 需 1 cycle 停顿）③ 控制冒险（分支误预测 1 cycle）。17.4% 的额外开销属于合理范围 |
| **为什么不用直接映射（1-way）？** | riscv-mini 默认 nWays=2。2-way 可以有效减少 conflict miss。本实验固定 nWays=2，变量为 nSets 和 blockBytes |
| **32B 块 miss 率降一半，为什么不直接推荐 c3？** | D$ miss 从 198→100 确实好，但 98 次 miss 的差异在 14 万 total cycle 中占比 <1%，CPI 无变化。对于 16KB 面积预算，c2 和 c3 面积相近（tag 开销不同），可根据具体访存模式选择 |
| **TIMEOUT 配置能不能估算 CPI？** | 不能。thrashing 状态下程序逻辑未推进，instret 计数器不增长（始终执行 NOP），CPI 趋于无穷大。5000 万 cycle 本身是有效的实验证据 |
| **为什么选矩阵乘法？** | ① 三重循环提供可调规模（N=4/8/16）② 内层 k 循环顺序访问 A（好空间局部性）+ 跨行访问 B（差空间局部性），能暴露 Cache 行为差异 ③ 软件乘法增加计算密度 |
| **Cache.stall 计数器为什么这样设计？** | 阅读 Cache.scala 源码：`stall_cnt` 在 `state =/= sIdle` 时自增。它衡量的是 Cache 利用率（busy time%），不是流水线停顿。I$ busy ≈ 98.7% 是预期的——取指几乎不停 |
