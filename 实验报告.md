## 一、实验概述

本实验基于 UC Berkeley 开源的 riscv-mini（RV32I 三级流水线处理器，Chisel 实现），围绕两个核心任务展开：

### 3.3 matrix_mul N=16 Cache 参数扫描（核心实验）

以自主编写的 16×16 矩阵乘法为测试负载：

| 配置 | 容量 | Cycles | CPI | I$ Miss% | D$ Miss% | 状态 |
|------|------|--------|-----|----------|----------|------|
| c2   | 16KB | 141,213 | 1.21 | 2.57 | 1.84 | ✅ PASS |
| c3   | 16KB | 141,108 | 1.21 | 1.22 | 0.93 | ✅ PASS |
| c1   | 8KB  | 50,000,000+ | — | — | — | ❌ TIMEOUT |
| c4   | 4KB  | 50,000,000+ | — | — | — | ❌ TIMEOUT |
| c5   | 4KB  | 50,000,000+ | — | — | — | ❌ TIMEOUT |

### 3.4 c2 vs c3 精细对比（同为 16KB，不同块大小）

| 指标 | c2 (512s, 16B块) | c3 (256s, 32B块) | 变化 |
|------|-------------------|-------------------|------|
| I$ Miss | 34 | 16 | ↓53% |
| I$ Miss Rate | 2.57% | 1.22% | ↓53% |
| D$ Miss | 198 | 100 | ↓49% |
| D$ Miss Rate | 1.84% | 0.93% | ↓49% |
| Total Cycles | 141,213 | 141,108 | — |
| CPI | 1.21 | 1.21 | — |

### 3.5 分析

**发现 1：存在"性能悬崖"（Performance Cliff）**

16KB 配置在 ~14 万 cycle 正常完成，8KB 及以下配置在 5000 万 cycle（超时阈值，约 354 倍于正常执行时间）后仍未退出。说明存在硬性的容量门槛——低于 16KB 时，matrix_mul N=16 的 ~4KB 工作集因 2-way 组相联的冲突缺失导致 thrashing。

**发现 2：块大小增大显著降低 miss 率，但 CPI 不变**

c3（32B 块）相比 c2（16B 块）：I$ Miss ↓53%，D$ Miss ↓49%。然而 CPI 均为 1.21。由 CPI=1.21 推算实际流水线停顿约 24,564 cycles（占总 cycle 的 17.4%），c3 节省的 ~100 次 D$ miss × 3~6 cycle ≈ 300~600 cycles，仅占总 cycle 的 0.2%~0.4%，无法在 CPI 上体现。

> **关键澄清**：Cache 模块的 `stall_cnt` 计数器（Cache.scala L331-333）统计的是 `state =/= sIdle` 的周期数，即 **Cache 忙碌周期**，而非流水线停顿周期。I$ 几乎每个周期都在取指（sReadCache 状态），因此 I$Stall ≈ 总 cycle（139,295/141,213 = 98.7%），但这不等于流水线停顿。

**发现 3：此时瓶颈已转移至流水线内部**

CPI=1.21 意味着额外开销仅 21%，其中 Cache miss 仅贡献约 2,000~3,000 cycles，其余来自数据冒险（RAW hazard 旁路延迟）和控制冒险（分支误预测 1 cycle 惩罚）。因此 c2→c3 的 miss 率改善无法降低 CPI。

**发现 4：multiply 对 Cache 不敏感**

multiply CPI 恒为 1.32，因为它是计算密集型程序（软件乘法循环），访存极少（D$ 访问仅 709 次）。

**发现 5：matrix_mul CPI 随 N 增大反而降低**

| N | CPI | 原因 |
|---|-----|------|
| N=4 | 1.31 | 函数调用/初始化固定开销占比大 |
| N=8 | 1.23 | 计算量增长摊薄固定开销 |
| N=16 | 1.21 | 最大规模，固定开销占比最小 |

## 四、自定义裸机程序

### 4.1 程序结构

```
custom-bmark/
├── main.c          (46 行) — C 主程序，初始化 3 个 16×16 矩阵，调用汇编函数
├── matrix_mul.S    (129行) — RV32I 汇编矩阵乘法，三重循环 + 移位加法乘法
├── crt.S           — 启动代码（_start 入口，初始化 sp）
├── test.ld         — 链接脚本（入口 0x200，栈顶 0x10000）
├── syscalls.c      — _exit 简易实现
└── Makefile        — 编译规则
```

### 4.2 RV32I 指令覆盖

| 类型 | 指令 | 用途 |
|------|------|------|
| R-type | add, sub, sltu, srl, and, or | 地址计算、乘法循环 |
| I-type | addi, lw, sw, slli, srli, andi, ori | 加载/存储矩阵元素、移位 |
| B-type | beq, bne, bltu, bge | 三重循环条件判断、乘法位检查 |
| J-type | jal, ret | 函数调用/返回 |
| U-type | lui, auipc | （链接脚本自动生成） |
| SYSTEM | csrw, csrr | mtohost 退出协议 |

### 4.3 工具链全流程

```
main.c + matrix_mul.S + crt.S + syscalls.c
       ↓ riscv32-unknown-elf-gcc -nostdlib -T test.ld
       main (ELF)
       ↓ elf2hex
       main.hex
       ↓ sbt run → Tile.sv → make verilator → VTile
       ./VTile main.hex → mtohost=0 (PASS)
```

### 4.4 关键问题与解决

| 问题 | 根因 | 解决 |
|------|------|------|
| 入口 _start 在 0x400 而非 0x200 | crt.o 链接顺序靠后 | test.ld 中 `*crt.o(.text)` 优先于 `*(.text)` |
| 汇编器不认识 mtohost 符号 | 自定义 CSR 名不在标准头文件 | 改为数字地址 `0x780` |
| make verilator 不重新链接 | Makefile 隐式规则检测 VTile 已存在 | 每次参数变更前 `rm -f VTile` |
| 超时配置性能计数器为空 | top.cc 超时分支未打印计数器 | 在报告中标注 "-" 并分析代码路径 |

## 五、指令级流水线分析

### 5.1 LW Miss 时序（D$ miss 场景）

以 matrix_mul 内层循环中的 `LW s7, 0(t1)` 为例：

```
Cycle 1: I1(ADDI) W-stage | I2(LW) E-stage → dcache.req.valid=1, miss!
Cycle 2-4: stall = !dcache.resp.valid → 整个流水线停顿
           D$ 走 sReadCache→sRefill→sIdle，从 NASTI 总线 refill 整个 cache line
Cycle 5: dcache.resp.valid=1 → stall 解除
         I2(LW) W-stage (s7 ← mem_data)
         I3(ADD) E-stage, I4(SLLI) F-stage
```

Miss 惩罚：3~6 cycle（clean block 约 3 cycle，dirty block 需额外 evict+writeback 约 6 cycle）。

### 5.2 BEQ 分支预测

riscv-mini 默认预测不跳转（PC+4）。实际跳转时需 flush 取指级（inst_kill），损失 1 cycle。

### 5.3 ADD + 旁路（Bypass/Forwarding）

Datapath.scala L122-126 实现 EX→EX 前递：
```scala
val rs1hazard = wb_en && rs1_addr.orR && (rs1_addr === wb_rd_addr)
val rs1 = Mux(wb_sel === WB_ALU && rs1hazard, ew_reg.alu, regFile.io.rdata1)
```
数据冒险在 EX 级通过 Mux 直接旁路 ALU 结果，无需停顿。

## 六、总结

1. **容量阈值效应**：matrix_mul N=16 的工作集 ~4KB，因 2-way 组相联的冲突缺失，实际需要 16KB 才能正常运行。8KB 及以下陷入 thrashing，超时于 50M cycles（354× 正常时间）。

2. **块大小优化**：32B 块相比 16B 块，miss 率降低约 50%，但 CPI 不变（1.21）——瓶颈已从 Cache miss 转移至流水线数据/控制冲突。

3. **Cache 计数器语义**：`perf.stall` 统计的是 Cache 状态机忙碌周期（非空闲状态），不等同于流水线停顿。I$ 忙碌周期 ≈ 总 cycle（98.7%），而实际流水线停顿仅 17.4%（由 CPI=1.21 推算）。

4. **自定义裸机程序**：175 行 C+汇编矩阵乘法，覆盖 RV32I 全部 6 类指令格式，成功打通 C/ASM→ELF→hex→Verilator 仿真→性能计数器采集的完整工具链。

5. **工程教训**：Makefile 隐式规则、Shell 转义、SSH 超时等工程细节对实验效率影响巨大。"环境可复现"是科学实验的基础。

### D. 性能计数器定义

| 计数器 | 含义 | 注意 |
|--------|------|------|
| `total_cycles` | 仿真总周期 | main_time/2 |
| `instructions` | 退休指令数（CSR.instret） | 不含 NOP/stall |
| `CPI` | total_cycles / instructions | 理想为 1.0 |
| `icache_stall` | **I$ 忙碌周期**（state≠sIdle） | ≠ 流水线停顿 |
| `dcache_stall` | **D$ 忙碌周期**（state≠sIdle） | ≠ 流水线停顿 |
| `icache_miss` | I$ miss 次数 | 触发 refill 的次数 |
| `dcache_miss` | D$ miss 次数 | 触发 refill 的次数 |
